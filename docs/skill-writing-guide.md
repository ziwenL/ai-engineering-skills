# Skill Writing Guide

一个标准 Skill 应包含：

```text
frontmatter:
  name
  description

正文：
  适用场景
  输入
  工作流程
  输出格式
  约束
  不适用场景
```

## 基本原则

1. 描述要具体，优先写清楚“什么时候该用这个 Skill”。
2. 流程要可执行，避免只有口号没有操作顺序。
3. 输出要结构化，便于复用和验收。
4. 约束要明确，尤其是不能做什么、缺信息时怎么处理。
5. 不要写入临时信息、个人路径或敏感内容。
6. 如果 Skill 目录下有 `checklist.md`、`architecture.md`、`log-analysis.md` 等辅助文件，必须在 `SKILL.md` 中明确要求读取和使用。
7. 不要只写“检查列表”，要写清楚先读什么、怎么做、最后怎么输出。
8. 开发类 Skill 必须写清验证闭环，至少说明需要运行的编译、测试、Lint 或人工验证方式。
9. 审查类 Skill 必须写清证据要求、风险分级和是否阻塞合并或发布。

## 推荐章节模板

```text
## 适用场景
## 输入
## 工作流程
## 输出格式
## 约束
## 不适用场景
```

## 调用示例约定

为了降低实际使用时的提示词组织成本，推荐为高频 Skill 增加一层调用示例。

推荐结构：

```text
skills/<skill-name>/
├── SKILL.md
└── usage-examples.md
```

其中：

- `SKILL.md` 负责放一个最小调用示例，保证用户打开就能复制使用。
- `usage-examples.md` 负责放更完整的多场景示例，避免主文档过长。

## `SKILL.md` 中的示例写法

建议在 `SKILL.md` 末尾增加一个简单区块：

````markdown
## 使用示例

最小调用示例：

```text
请使用 xxx skill ...
```

更多示例见 `usage-examples.md`。
````

注意：

- 如果目录下存在 `usage-examples.md`，必须在 `SKILL.md` 中显式引用它。
- 最小调用示例要短、可复制、可直接执行。
- 不要把所有变体都塞进 `SKILL.md`。

## `usage-examples.md` 推荐结构

```text
# 使用示例
## 说明
## 最小调用示例
## 推荐调用示例
## 高约束调用示例
## 适用提醒
```

## 示例写作要求

调用示例应尽量是“完整提示词”，而不是解释性描述。

推荐包含：

- 明确指定 Skill 名称
- 先阅读哪些文件或上下文
- 需要输出什么结构
- 不能做什么或必须遵守什么约束

避免：

- 只有一句“请处理这个需求”
- 不说明输出结构
- 让模型自行猜测项目事实
- 让 Skill 去承担它本不负责的工作
