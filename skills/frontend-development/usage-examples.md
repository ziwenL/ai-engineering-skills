# 使用示例

## 说明

这个 Skill 适合在 Web、H5、管理后台或组件开发中，按现有项目模式实现页面、组件、状态管理和接口接入，并覆盖关键状态与验证。

## 最小调用示例

```text
请使用 frontend-development skill 实现该需求。

请先阅读：
1. 已确认的需求分析
2. 已确认的技术方案
3. 项目入口文档
4. skills/common/
5. 相似页面、组件和状态管理实现

要求：
- 先列出计划修改的文件和原因。
- 不要只实现 happy path。
- 实现后运行构建、Lint 或测试命令。

最后输出：
1. 修改摘要
2. 验证结果
3. 未验证项
4. 风险点
```

## 推荐调用示例

```text
请使用 frontend-development skill 在当前项目中实现这个需求。

请按以下顺序进行：
1. 阅读需求分析和技术方案
2. 阅读项目入口文档和 skills/common/
3. 查找相似页面、组件和状态管理方式
4. 先列出计划修改的文件和每个文件的修改原因

实现要求：
1. 遵守当前项目结构和状态管理方式
2. 不引入新依赖，除非先明确说明必要性
3. 覆盖 loading / empty / error / success
4. 必要时覆盖 retry、permission denied、timeout
5. 检查埋点、兼容性、可访问性和回归影响
6. 不做与当前需求无关的重构

完成后请输出：
1. 修改文件列表
2. 核心实现逻辑
3. 状态与异常处理
4. 接口接入变化
5. 验证命令与结果
6. 未验证项
7. 风险点
8. 人工验证建议
```

## 高约束调用示例

```text
请使用 frontend-development skill 实现当前需求，并严格保持最小改动原则。

必须先阅读：
1. 已确认的需求分析
2. 已确认的技术方案
3. 项目入口文档
4. skills/common/
5. 相似页面、组件和状态管理实现

执行要求：
- 先输出计划修改文件清单，不要直接开改。
- 不要只实现 happy path。
- 如果无法运行验证命令，要明确说明原因。

请严格按以下结构输出：
【修改文件列表】
【每个文件的修改原因】
【核心实现逻辑】
【状态与异常处理】
【接口接入变化】
【验证命令与结果】
【未验证项】
【风险点】
【人工验证建议】
```

## 适用提醒

- 适合 Web / H5 / 管理后台等已有前端项目内的开发与修复。
- 如果项目本身还没起盘，应先走相应的起盘或 scaffold 路径。
- 如果需求和方案还没确认，应先用 `requirement-analysis` 和 `technical-design`。
