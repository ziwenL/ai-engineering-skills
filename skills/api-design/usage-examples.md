# 使用示例

## 说明

这个 Skill 适合在服务端实现前，先把接口路径、请求参数、响应结构、错误码、鉴权、兼容性和验证方式设计清楚。

## 最小调用示例

```text
请使用 api-design skill 设计这个接口。

请输出：
1. 接口目标
2. 请求方法与路径
3. 请求参数
4. 响应结构
5. 错误码
6. 兼容性
7. 验证方式
8. 未验证项

要求：
- 不要直接进入服务端实现。
- 不要只给 happy path。
```

## 推荐调用示例

```text
请使用 api-design skill，为当前需求设计接口契约。

请先明确：
1. 业务目标和调用场景
2. 调用方类型
3. 已有相似接口和兼容约束

请输出：
1. 接口目标
2. 请求方法与路径
3. 请求参数
4. 响应结构
5. 错误码
6. 鉴权与权限
7. 幂等与限流
8. 兼容性
9. 示例
10. 验证方式
11. 未验证项

要求：
- 不要只给 happy path，要覆盖错误码和异常场景。
- 如果影响旧客户端，必须写清兼容策略。
- 无法确认的字段语义要标记为待确认或未验证项。
```

## 高约束调用示例

```text
请使用 api-design skill，严格设计当前接口契约。

请严格按以下结构输出：
【接口目标】
【请求方法与路径】
【请求参数】
【响应结构】
【错误码】
【鉴权与权限】
【幂等与限流】
【兼容性】
【示例】
【验证方式】
【未验证项】

要求：
- 不要直接进入 backend-development。
- 不要只给 happy path。
- 没有验证路径时，不要把设计写成“已确认无风险”。
```

## 适用提醒

- 适合接口边界仍需设计和对齐的阶段。
- 如果当前只是内部实现调整且无接口变化，不一定需要它。
- 如果接口设计已经稳定，下一步通常接 `backend-development` 或 `backend-service-scaffold`。
