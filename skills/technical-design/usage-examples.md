# 使用示例

## 说明

这个 Skill 适合在需求已经相对明确后，用来生成可执行的技术方案。重点不是直接编码，而是先把改动范围、模块影响、验证方式和风险说清楚。

## 最小调用示例

```text
请基于已确认的需求分析，使用 technical-design skill 生成技术方案。

请先：
1. 阅读需求分析结论
2. 查找项目中的相似实现
3. 结合当前架构与模块边界进行设计

请输出：
1. 总体实现思路
2. 涉及模块
3. 修改文件清单
4. 数据结构或接口变化
5. 状态流转
6. 错误处理
7. 测试方案
8. 风险点

要求：
- 不要直接开始改代码。
- 明确区分已确认事实与假设。
```

## 推荐调用示例

```text
请使用 technical-design skill，为这个需求生成一份最小可行的技术方案。

请先完成以下上下文准备：
1. 阅读已经确认的需求分析
2. 查找项目中的相似实现
3. 理解当前架构边界、状态管理方式和依赖约束

请输出：
1. 总体方案
2. 相似实现参考
3. 涉及模块
4. 修改文件清单
5. 数据结构变化
6. 接口变化
7. 状态流转
8. 错误处理与回滚思路
9. 测试方案
10. 验证方式
11. 风险与兼容性
12. 未验证项

要求：
- 优先最小改动路径，不要顺手重构。
- 对无法确认的地方明确标记为假设或未验证项。
- 如果需要多端联动，请明确分别影响哪些端。
```

## 高约束调用示例

```text
请使用 technical-design skill，为当前需求生成技术方案，并严格控制方案范围。

请先完成：
1. 阅读需求分析结论
2. 查找至少 1 个项目内相似实现
3. 确认当前模块边界与依赖关系

请严格按以下结构输出：
【总体方案】
【相似实现】
【涉及模块】
【修改文件清单】
【数据结构】
【接口变化】
【状态流转】
【错误处理】
【测试方案】
【验证方式】
【风险与兼容】
【是否需要灰度 / 开关】
【未验证项】

要求：
- 不要进入编码阶段。
- 不要把推测写成事实。
- 如果验证路径不清晰，必须写进【未验证项】。
- 如果方案过大，请主动收敛成最小可行版本。
```

## 适用提醒

- 适合“需求已基本确认，但还没开始实现”的阶段。
- 如果需求本身还没分析清楚，应该先用 `requirement-analysis`。
- 如果你已经有成熟技术方案，要直接编码，则切到对应开发 Skill。
