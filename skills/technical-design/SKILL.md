# 技术方案 Skill

## 适用场景

- 需求分析确认后的技术方案设计
- 复杂改动前的模块影响和风险评估
- 需要先明确文件改动、接口变化和验证方式

## 输入

- 已确认的需求分析
- 项目中相似实现
- 当前架构和模块边界

## 工作流程

1. 查找项目中相似实现，确认可复用路径。
2. 判断涉及模块、数据流和状态流转。
3. 设计错误处理、回滚路径和兼容策略。
4. 评估接口和数据结构变化。
5. 设计测试方案和验证方式。
6. 区分已确认事实、关键假设和需要灰度/开关支持的点。
7. 输出最小可行实现路径、风险点和未验证项。

## 输出格式

```text
【总体方案】
【相似实现】
【涉及模块】
【修改文件清单】
【数据结构】
【接口变化】
【状态流转】
【错误处理】
【测试方案】
【验证方式】
【风险与兼容】
【是否需要灰度/开关】
【未验证项】
```

## 约束

- 不要直接进入代码实现。
- 明确哪些是事实，哪些是假设。
- 优先最小可行方案，避免无关重构。
- 如果验证路径不清晰，要明确写进“未验证项”。

## 不适用场景

- 需求还没有分析清楚
- 只是很小的纯文案改动，不需要独立技术方案
