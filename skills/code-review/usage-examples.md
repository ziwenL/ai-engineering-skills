# 使用示例

## 说明

这个 Skill 适合在改动完成后做审查，重点是找真实风险、证据和测试缺口，而不是简单复述 diff。

## 最小调用示例

```text
请使用 code-review skill 审查本次改动。

请重点检查：
1. 需求符合性
2. 逻辑正确性
3. 边界条件
4. 测试缺口
5. 安全 / 性能 / 可维护性风险

请输出：
1. 主要问题
2. 证据
3. 风险等级
4. 是否阻塞合并
```

## 推荐调用示例

```text
请使用 code-review skill 审查当前改动。

请先阅读：
1. 任务背景
2. 需求分析
3. 技术方案
4. 代码 diff
5. 已有测试结果

请重点关注：
1. 改动是否符合需求
2. 是否破坏现有架构约束
3. 是否遗漏错误处理、边界条件或回归风险
4. 是否存在安全、性能或可维护性问题
5. 测试覆盖是否足够

请输出：
1. 结论
2. 主要问题
3. 次要问题
4. 证据
5. 风险等级
6. 测试缺口
7. 风险点
8. 建议修改
9. 是否阻塞合并

要求：
- 不要只复述改动内容。
- 如果证据不足，请明确写成待确认。
- 高风险问题优先于样式问题。
```

## 高约束调用示例

```text
请使用 code-review skill 对当前 diff 进行严格审查。

必须基于以下输入：
1. 任务背景
2. 需求分析
3. 技术方案
4. 代码 diff
5. 测试结果

请严格按以下结构输出：
【结论】
【主要问题】
【次要问题】
【证据】
【风险等级】
【测试缺口】
【风险点】
【建议修改】
【是否阻塞合并】

要求：
- 不能只复述改动，要指出真实风险和缺口。
- 如果没有证据支持，不要下结论。
- 阻塞判断必须与证据和风险等级对应。
```

## 适用提醒

- 适合在实现完成后、合并前使用。
- 如果还没有 diff，这个 Skill 不适合直接使用。
- 如果你要的是“直接修改代码”而不是先审查，应切到对应开发 Skill。
