# 代码审查 Skill

## 适用场景

- 审查本次 diff
- 检查需求符合性和架构一致性
- 识别安全、性能、测试和可维护性问题

## 输入

- 任务背景
- 需求分析和技术方案
- 代码 diff
- 相关测试结果

## 工作流程

1. 阅读任务背景、需求分析和技术方案。
2. 阅读 diff，确认改动范围和主要行为变化。
3. 对照需求、架构和测试结果检查逻辑正确性。
4. 优先识别高风险问题，例如错误处理缺失、边界条件遗漏、测试缺口、安全/性能回归。
5. 对每个主要问题给出证据、风险等级和建议修改方向。
6. 判断这些问题是否阻塞合并。

## 输出格式

```text
【结论】
【主要问题】
【次要问题】
【证据】
【风险等级】
【测试缺口】
【风险点】
【建议修改】
【是否阻塞合并】
```

## 约束

- 不能只复述改动，要指出真正的风险和缺口。
- 如果证据不足，要明确写成待确认，不要硬下结论。
- 高风险问题要优先于样式问题。
- 阻塞判断要和证据、风险等级对应。

## 不适用场景

- 没有 diff 的纯需求讨论
- 需要直接实施改动而不是先审查的场景
