# 使用示例

## 说明

这个 Skill 适合在监控告警刚出现时，先判断异常指标、开始时间、影响范围和可能关联的变更，不直接跳进代码修复。

## 最小调用示例

```text
请使用 monitoring-analysis skill 分析这个监控异常。

已知信息：
1. 告警内容
2. 指标或图表
3. 时间范围
4. 最近发布和配置变更

请输出：
1. 异常指标
2. 开始时间
3. 影响范围
4. 拆分维度
5. 已确认事实
6. 根因假设
7. 下一步排查建议

要求：
- 不要只看单一指标下结论。
- 必须区分相关性和因果性。
```

## 推荐调用示例

```text
请使用 monitoring-analysis skill 分析当前监控异常。

请基于以下输入：
1. 告警信息
2. 监控指标或图表
3. 异常时间范围
4. 最近发布记录
5. 最近配置变更

请输出：
1. 异常指标
2. 开始时间
3. 影响范围
4. 拆分维度
5. 已确认事实
6. 根因假设
7. 下一步排查建议

要求：
- 按版本、平台、地区、渠道、机型或流量维度拆分。
- 不要把相关性直接写成因果结论。
- 如果数据不足，明确写出还需要补哪些监控或日志。
```

## 高约束调用示例

```text
请使用 monitoring-analysis skill，严格分析当前监控异常。

请严格按以下结构输出：
【异常指标】
【开始时间】
【影响范围】
【拆分维度】
【已确认事实】
【根因假设】
【下一步排查建议】

要求：
- 不允许只看单一指标就下结论。
- 必须区分事实、相关性和因果推断。
- 数据不足时，直接写明缺口。
```

## 适用提醒

- 适合监控刚发现异常时使用。
- 如果已经进入具体日志比对和问题定位，下一步通常接 `bug-triage`。
- 这个 Skill 不是直接修代码的入口。
