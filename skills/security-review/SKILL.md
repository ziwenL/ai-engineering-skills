# 安全审查 Skill

## 适用场景

- 代码改动后的安全审查
- 发版前高风险模块检查
- 涉及鉴权、权限、输入处理、日志和敏感信息的改动

## 输入

- 代码 diff 或关键文件
- 需求背景和接口边界
- 已知风险点
- 相关配置、日志或测试结果

## 工作流程

1. 明确改动边界、调用方和信任边界。
2. 检查输入、输出、鉴权、权限和敏感操作路径。
3. 检查密钥、Token、日志、隐私、注入、越权和最小权限问题。
4. 区分已确认问题、待验证风险和低置信度猜测。
5. 为每个问题给出证据、风险等级、影响范围和修复建议。
6. 明确是否阻塞合并或发版。

## 输出格式

```text
【审查结论】
【主要风险】
【风险等级】
【受影响文件】
【证据】
【修复建议】
【待确认问题】
【是否阻塞合并/发版】
```

## 约束

- 没有证据时不要伪装成确定漏洞。
- 高风险问题要优先于样式和低影响问题。
- 不能默认“没有看到问题”就等于“绝对安全”。

## 不适用场景

- 没有代码或行为上下文的纯空泛咨询
- 需要真正渗透测试或专业攻防验证的场景

## 检查项

1. 密钥 / Token / 证书是否泄露。
2. 是否存在越权访问。
3. 是否存在 SQL 注入、XSS、命令注入。
4. 是否绕过权限或鉴权。
5. 日志是否包含隐私数据。
6. 客户端是否硬编码敏感信息。
7. 服务端是否缺少限流和审计。
8. 数据库访问是否遵循最小权限。
