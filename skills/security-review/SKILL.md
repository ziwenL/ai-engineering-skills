# 安全审查 Skill

## 适用场景

- 代码改动后的安全审查
- 发布前高风险模块检查
- 涉及鉴权、权限、输入处理、日志和敏感信息的改动

## 输入

- 代码 diff 或关键文件
- 需求背景和接口边界
- 已知风险点
- 相关配置、日志或测试结果
- `usage-examples.md`

## 工作流程

1. 明确改动边界、调用方和信任边界。
2. 检查输入、输出、鉴权、权限和敏感操作路径。
3. 检查密钥、Token、日志、隐私、注入、越权和最小权限问题。
4. 区分已确认问题、待验证风险和低置信度猜测。
5. 为每个问题给出证据、风险等级、影响范围和修复建议。
6. 明确是否阻塞合并或发布。

## 输出格式

```text
【审查结论】
【主要风险】
【风险等级】
【受影响文件】
【证据】
【修复建议】
【待确认问题】
【是否阻塞合并 / 发布】
```

## 约束

- 没有证据时不要伪装成确定漏洞。
- 高风险问题要优先于样式和低影响问题。
- 不能默认“没有看到问题”就等于“绝对安全”。

## 不适用场景

- 没有代码或行为上下文的纯空泛咨询。
- 需要真正渗透测试或专业攻防验证的场景。

## 使用示例

最小调用示例：

```text
请使用 security-review skill 审查本次改动。

请重点关注：
1. 密钥 / Token / 证书泄露
2. 越权访问
3. 注入风险
4. 权限或鉴权绕过
5. 日志中的隐私数据

请输出：
1. 审查结论
2. 主要风险
3. 风险等级
4. 证据
5. 修复建议
6. 是否阻塞合并 / 发布
```

更多示例见 `usage-examples.md`。
