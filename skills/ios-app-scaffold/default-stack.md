# iOS App Scaffold 默认技术栈

## 推荐默认方案

- Swift
- SwiftUI
- 单 Xcode 工程起步，不默认拆分 `workspace`、多 `target` 或本地 `Swift Package`
- 目录优先按最小职责分层组织：`App`、`Features`、`Core`、`Services`、`Resources`
- 第一阶段只要求最小可运行首页、应用入口、基础主题、导航占位和网络服务占位

## 备选方案

- 如果项目已明确要求沿用 UIKit 体系，切换到 `UIKit + ViewModel` 风格
- 如果项目第一阶段就存在强共享组件、强包边界或多团队并行开发约束，可升级到 `workspace + package/target` 方案
- 如果首个里程碑仅用于本地演示且没有真实接口依赖，可暂时弱化网络实现，只保留服务协议和 mock 占位

## 何时保持默认方案

- 团队没有明确要求放弃 SwiftUI，且首批页面适合声明式 UI
- 第一阶段重点是尽快得到可编译、可运行、可继续扩展的 iOS 工程壳子
- 当前还没有证据证明必须上多 target、workspace 或复杂模块边界
- 需要先把页面层、状态层、服务层的最小边界稳定下来，再进入功能开发

## 何时切换到备选方案

- 团队已有稳定 UIKit 基础设施、模板和代码审查规范，继续使用 SwiftUI 的迁移成本更高
- 已确认第一阶段就要并行开发多个独立能力，且单工程结构无法支撑边界隔离
- 需要复用既有内部包、二进制依赖或严格 target 配置，单工程默认方案无法平滑接入
- 产品演示阶段完全依赖本地假数据，且短期内不会接入真实服务

## 选择原则

- 第一批优先保证最小可运行、可验证、可扩展
- 没有明确约束时，不提前引入复杂 `target`、`workspace` 或依赖拆分
- 任何切换都要说明切换条件、收益、代价，以及对后续 `ios-development` 的影响
