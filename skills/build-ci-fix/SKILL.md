# 构建与 CI 修复 Skill

## 适用场景

- 编译失败
- 测试失败
- Lint 或格式检查失败
- CI 环境和本地环境不一致

## 输入

- 失败日志
- 最近相关改动
- 本地可运行的对应命令
- `usage-examples.md`

## 工作流程

1. 阅读失败日志，定位失败阶段：编译 / 测试 / Lint / 打包 / 发布。
2. 提取最小错误信息，确认是代码问题、依赖问题还是环境差异。
3. 在本地复现失败命令，验证失败现象和错误原因。
4. 查找相关文件、最近改动和 CI 特有配置差异。
5. 给出最小修复方案，避免顺手做无关重构。
6. 重新运行失败命令，记录验证结果。
7. 标记尚未复现、尚未验证或仍依赖远端环境确认的未验证项。

## 输出格式

```text
【失败阶段】
【关键错误】
【根因假设】
【最小修复方案】
【验证命令】
【验证结果】
【未验证项】
【风险点】
```

## 约束

- 不允许为了通过 CI 删除测试。
- 不允许随意降低 Lint 或检查规则。
- 没有复跑验证前，不要声称“CI 已修复”。
- 如果只能部分验证，要明确写进“未验证项”。

## 不适用场景

- 没有日志和失败命令的模糊问题。
- 与构建无关的纯业务需求开发。

## 使用示例

最小调用示例：

```text
请使用 build-ci-fix skill 分析这段 CI 失败日志，并给出最小修复方案。

请输出：
1. 失败阶段
2. 关键错误
3. 根因假设
4. 最小修复方案
5. 验证命令
6. 验证结果
7. 未验证项

要求：
- 不要为了通过 CI 删除测试。
- 不要随意降低 Lint 或检查规则。
```

更多示例见 `usage-examples.md`。
