# 使用示例

## 说明

这个 Skill 适合在需求分析和技术方案明确后生成测试用例，重点是覆盖正向、异常、边界、兼容和回归，而不是只写 happy path。

## 最小调用示例

```text
请使用 test-case-generation skill，基于需求分析和技术方案生成测试用例。

请覆盖：
1. 正向用例
2. 异常用例
3. 边界用例
4. 兼容性用例
5. 回归用例
6. 自动化建议
7. 人工验证建议

要求：
- 不要只覆盖 happy path。
- 无法确认的行为请写入未验证项。
```

## 推荐调用示例

```text
请使用 test-case-generation skill，为当前需求输出一组可执行的测试用例建议。

请先阅读：
1. 需求分析结论
2. 技术方案
3. 已知兼容性限制
4. 已知风险点

请输出：
1. 正向用例
2. 异常用例
3. 边界用例
4. 兼容性用例
5. 回归用例
6. 自动化建议
7. 人工验证建议
8. 未验证项

要求：
- 区分哪些适合自动化，哪些更适合人工验证。
- 覆盖主流程、异常流程和边界输入。
- 如果涉及权限、网络、超时、多端一致性或版本兼容，请明确补充相关场景。
```

## 高约束调用示例

```text
请使用 test-case-generation skill，基于当前需求分析和技术方案生成测试用例。

请严格按以下结构输出：
【正向用例】
【异常用例】
【边界用例】
【兼容性用例】
【回归用例】
【自动化建议】
【人工验证建议】
【未验证项】

要求：
- 不要只写 happy path。
- 自动化建议和人工验证建议要明确区分。
- 对需求未确认部分，不要伪装成已确定测试用例，直接写进【未验证项】。
```

## 适用提醒

- 适合在需求和方案已经比较稳定后使用。
- 如果需求尚未分析清楚，应先使用 `requirement-analysis`。
- 如果技术方案还不明确，测试用例也会缺少稳定基础。
