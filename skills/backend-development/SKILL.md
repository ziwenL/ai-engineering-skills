# 服务端开发 Skill

## 适用场景

- 新接口或新业务逻辑开发
- 现有接口行为修复
- 服务端数据结构、错误码、兼容性调整
- 需要同时考虑鉴权、权限、幂等、限流和监控的改动

## 输入

- 已确认的需求分析
- 已确认的技术方案
- 相似接口和相似业务逻辑
- 项目入口文件和 `skills/common/`
- `api-checklist.md`
- `security-checklist.md`
- `usage-examples.md`

## 工作流程

1. 阅读需求分析和技术方案。
2. 阅读 `api-checklist.md`，明确接口契约最少需要覆盖哪些点。
3. 阅读 `security-checklist.md`，先识别权限边界、敏感日志和 Token 校验风险。
4. 查找相似接口和相似业务逻辑。
5. 明确请求链路、入参、出参、错误码和兼容边界。
6. 明确鉴权、权限、幂等、限流、日志和监控影响。
7. 先列出计划修改文件和原因，再实现最小可行改动。
8. 补充必要测试，优先覆盖主流程、异常流程和回归风险。
9. 运行相关测试或验证命令完成验证。
10. 输出兼容性、回滚方式、验证结果、未验证项和风险点。

## 输出格式

```text
【修改文件列表】
【每个文件的修改原因】
【接口契约变化】
【核心业务逻辑】
【鉴权 / 权限 / 幂等 / 限流检查】
【日志与监控影响】
【测试与验证结果】
【兼容性与回滚方案】
【未验证项】
【风险点】
```

## 约束

- 不做无关重构。
- 不引入新依赖，除非先得到确认。
- 必须使用 `api-checklist.md` 和 `security-checklist.md` 做自检。
- 不能修改生产配置或生产数据。
- 没有验证证据时，不能声称接口已可用。

## 不适用场景

- 纯前端或纯移动端任务。
- 尚未确认需求或方案的高风险改动。
- 需要先做人工批准的生产变更。

## 使用示例

最小调用示例：

```text
请使用 backend-development skill 实现该需求。

请先阅读：
1. 已确认的需求分析
2. 已确认的技术方案
3. 项目入口文档
4. skills/common/
5. skills/backend-development/api-checklist.md
6. skills/backend-development/security-checklist.md
7. 服务端相似接口或业务实现

要求：
- 先列出计划修改的文件和原因。
- 明确接口变化、错误处理和兼容性影响。
- 完成后运行相关测试或验证命令。

最后输出：
1. 修改摘要
2. 验证结果
3. 未验证项
4. 风险点
```

更多示例见 `usage-examples.md`。
