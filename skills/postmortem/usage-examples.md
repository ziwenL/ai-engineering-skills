# 使用示例

## 说明

这个 Skill 适合在线上事故结束后做复盘，把时间线、根因、处理过程、短期改进和长期防线整理清楚。

## 最小调用示例

```text
请使用 postmortem skill，基于本次事故生成复盘报告。

请输出：
1. 事故摘要
2. 影响范围
3. 时间线
4. 根因分析
5. 处理过程
6. 做得好的地方
7. 问题与不足
8. 短期改进
9. 长期防线
10. 需要沉淀到 Skills 的内容

要求：
- 不要甩锅，聚焦系统改进。
- 没有证据时不要把假设写成确定根因。
```

## 推荐调用示例

```text
请使用 postmortem skill，对这次线上事故做复盘。

请基于以下输入：
1. 事故时间线
2. 影响范围
3. 止血和修复过程
4. 已确认事实和证据

请输出：
1. 事故摘要
2. 影响范围
3. 时间线
4. 根因分析
5. 处理过程
6. 做得好的地方
7. 问题与不足
8. 短期改进
9. 长期防线
10. 需要沉淀到 Skills 的内容

要求：
- 不甩锅，聚焦系统性改进。
- 区分已确认事实、直接原因和系统性原因。
- 改进行动要尽量可执行，不要停留在口号。
```

## 高约束调用示例

```text
请使用 postmortem skill，严格输出本次事故复盘。

请严格按以下结构输出：
【事故摘要】
【影响范围】
【时间线】
【根因分析】
【处理过程】
【做得好的地方】
【问题与不足】
【短期改进】
【长期防线】
【需要沉淀到 Skills 的内容】

要求：
- 不甩锅，聚焦系统改进。
- 没有证据时不要写成确定根因。
- 短期改进和长期防线都要可执行。
```

## 适用提醒

- 适合事故已经结束后使用。
- 如果事故还在处理中，应先用 `incident-response`。
- 如果复盘里有可复用经验，后续通常接 `session-retrospective` 或 `skill-authoring`。
