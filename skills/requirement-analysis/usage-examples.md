# 使用示例

## 说明

这个 Skill 适合在需求刚进入开发流程时使用，先把业务目标、用户故事、主流程、异常流程和待确认项拆清楚，再进入技术方案或实现阶段。

## 最小调用示例

```text
请使用 requirement-analysis skill 分析以下需求。

请输出：
1. 业务目标
2. 用户故事
3. 主流程
4. 异常流程
5. 边界条件
6. 依赖模块
7. 待确认问题
8. 初步风险

要求：
- 不要写代码。
- 不要直接进入技术方案。
- 对无法确认的信息标记为“待确认”。
```

## 推荐调用示例

```text
请使用 requirement-analysis skill 分析这个需求，并先把业务问题拆清楚，再进入后续设计阶段。

请基于以下输入进行分析：
1. 原始需求描述
2. 当前业务背景
3. 已知限制条件
4. 可能依赖的模块或外部系统

请输出：
1. 业务目标
2. 用户故事
3. 主流程
4. 异常流程
5. 边界条件
6. 涉及模块
7. 待确认问题
8. 初步风险点

要求：
- 不要把猜测写成事实。
- 不要直接输出技术实现方案。
- 如果需求本身冲突或缺信息，要明确指出。
```

## 高约束调用示例

```text
请使用 requirement-analysis skill 对当前需求做结构化分析。

请严格按以下结构输出：
【业务目标】
【用户故事】
【主流程】
【异常流程】
【边界条件】
【依赖模块】
【待确认问题】
【初步风险】

要求：
- 不要写代码。
- 不要直接进入 technical-design。
- 所有无法确认的信息必须写入【待确认问题】。
- 如果需求描述中包含多个目标，请先拆分再分析。
```

## 适用提醒

- 适合“需求还没有被拆清楚”的阶段。
- 如果业务和产品约束已经很明确，下一步通常接 `technical-design`。
- 这个 Skill 的目标是澄清问题，不是生成实现方案。
