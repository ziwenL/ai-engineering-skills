# 性能审查 Skill

## 适用场景

- 代码改动后的性能风险审查
- 发版前性能重点项检查
- Android、iOS、Web、服务端的性能问题初筛

## 输入

- 代码改动或 diff
- 相关页面、接口、模块
- 可用的性能数据、指标或测试结果

## 工作流程

1. 明确改动范围和性能敏感路径。
2. 判断属于 Android、iOS、Web 还是服务端场景。
3. 检查主线程阻塞、慢查询、接口耗时、资源加载、内存、包体积等风险点。
4. 区分已有证据和基于经验的风险假设。
5. 给出每个主要问题的证据、风险等级、影响范围和验证方法。
6. 给出优化建议，并判断是否阻塞发版。

## 输出格式

```text
【性能风险点】
【风险等级】
【影响范围】
【证据】
【验证方法】
【优化建议】
【是否阻塞发版】
```

## 约束

- 没有数据时不能伪装成已验证结论。
- 不要只凭感觉要求复杂优化。
- 需要明确哪些问题是高概率风险，哪些只是待验证假设。
- 阻塞发版判断要和风险等级、证据匹配。

## 不适用场景

- 没有任何性能敏感路径的纯文档改动
- 已经进入详细性能实验阶段、需要专门 profiling 的任务

## 检查项

- Android 主线程阻塞
- iOS 卡顿
- Web 首屏和资源加载
- 服务端慢查询
- N+1 查询
- 图片加载
- 内存泄漏
- 接口耗时
- 启动性能
- 包体积
