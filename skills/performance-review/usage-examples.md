# 使用示例

## 说明

这个 Skill 适合在改动后做性能风险审查，重点是识别主线程阻塞、慢查询、接口耗时、资源加载和内存等风险，并区分证据与假设。

## 最小调用示例

```text
请使用 performance-review skill 审查本次改动。

请输出：
1. 性能风险点
2. 风险等级
3. 影响范围
4. 证据
5. 验证方法
6. 优化建议
7. 是否阻塞发布
```

## 推荐调用示例

```text
请使用 performance-review skill 审查当前改动的性能风险。

请基于以下输入：
1. 代码改动或 diff
2. 相关页面、接口或模块
3. 可用的性能数据、指标或测试结果

请输出：
1. 性能风险点
2. 风险等级
3. 影响范围
4. 证据
5. 验证方法
6. 优化建议
7. 是否阻塞发布

要求：
- 先判断属于 Android、iOS、Web 还是服务端场景。
- 区分已有证据和基于经验的风险假设。
- 不要只凭感觉要求复杂优化。
```

## 高约束调用示例

```text
请使用 performance-review skill，对当前改动做严格性能审查。

请严格按以下结构输出：
【性能风险点】
【风险等级】
【影响范围】
【证据】
【验证方法】
【优化建议】
【是否阻塞发布】

要求：
- 没有数据时不能伪装成已验证结论。
- 必须区分高概率风险和待验证假设。
- 阻塞发布判断要和风险等级、证据匹配。
```

## 适用提醒

- 适合改动后、发布前、或遇到明显性能敏感路径时使用。
- 如果已经进入详细 profiling 或性能实验阶段，这个 Skill 主要用于前置风险筛查。
- 如果只是普通文案改动，通常不需要它。
