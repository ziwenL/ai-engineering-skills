# Backend Service Scaffold 默认技术栈

## 推荐默认方案

- 单服务起步
- 轻量 REST API 风格
- 优先沿用团队已确认的主语言与主框架
- 如果语言和框架尚未确认，默认后备路线使用 `TypeScript + Node.js LTS + Fastify + Vitest`
- 在默认后备路线下，首批文件可直接按 `package.json`、`tsconfig.json`、`src/app.ts`、`src/server.ts`、`src/routes/health.ts`、`src/config/env.ts`、`src/plugins/error-handler.ts`、`src/plugins/logger.ts`、`tests/health.spec.ts` 展开
- 目录先按最小职责分层组织，例如 `app`、`config`、`http`、`service`、`domain`、`repository`、`tests`
- 第一阶段默认包含配置加载、健康检查、基础路由、统一错误处理、结构化日志和首批测试边界

## 备选方案

- 如果项目已明确要求更强分层，可增加 `handler`、`usecase`、`repository` 等更细边界
- 如果首个里程碑只是本地联调或演示服务，可暂时弱化持久化实现，仅保留接口占位和 mock 数据
- 如果团队已有严格约束的框架脚手架、配置中心、日志组件或测试模板，可切换到团队标准方案
- 如果团队明确以 `Python + FastAPI + pytest`、`Java/Kotlin + Spring Boot + JUnit` 或其他既有主栈为标准，可直接切换到团队标准后端路线
- 如果第一阶段已明确存在高并发、异步任务或事件驱动边界，可预留队列或 worker 占位，但不默认拆分成独立服务

## 何时保持默认方案

- 目标是先得到可启动、可验证、可继续扩展的最小服务壳子
- 当前还没有证据证明必须从第一天就拆分多服务或引入复杂基础设施
- 客户端、接口和业务边界仍在第一阶段收敛中，过早细拆会放大维护成本
- 团队暂时更需要稳定的本地开发、联调和测试入口，而不是复杂平台能力
- 团队虽然还没确认主栈，但接受先按明确的后备基线起服务壳子，后续再做受控迁移

## 何时切换到备选方案

- 团队已经有必须沿用的服务模板、框架约束或部署规则，默认方案无法平滑接入
- 接口契约、错误码体系、审计要求或安全要求已经证明需要更细的分层边界
- 第一阶段就必须接入数据库、缓存、消息队列、鉴权网关或其他强基础设施约束
- 里程碑目标只是临时演示或 mock 联调，正式持久化和正式依赖尚未进入本阶段

## 选择原则

- 第一批优先保证最小可运行、可验证、可扩展
- 没有明确约束时，不提前做微服务拆分、平台化抽象或复杂部署编排
- 没有团队既有偏好时，也必须落到一个明确后备基线，不能只停留在抽象选型原则
- 任何切换都要说明切换条件、收益、代价，以及对后续 `backend-development` 的影响
