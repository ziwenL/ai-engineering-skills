# Backend Service Scaffold Skill

## 适用场景

- 需要从 0 创建服务端新项目
- 需要为第一阶段里程碑搭建最小可运行的服务骨架
- 需要明确默认方案、备选方案与切换条件
- 需要输出最小服务结构、首批目录与文件、配置、健康检查、错误处理、日志和首批测试边界
- 需要为后续 `backend-development` 建立稳定落点

## 输入

- 已确认的产品目标、第一阶段范围和交付里程碑
- 已确认的技术方案、接口契约、错误码约定和客户端依赖边界
- 项目入口文件和 `skills/common/`
- 团队已知的语言、框架、部署环境、配置管理、日志和测试偏好
- 是否需要数据库、缓存、消息队列、对象存储、鉴权或第三方服务接入
- `scaffold-checklist.md`
- `default-stack.md`
- `usage-examples.md`

## 工作流程

1. 阅读项目入口文件和 `skills/common/`，确认当前任务是创建服务端新项目骨架，而不是继续修改已有服务。
2. 阅读 `scaffold-checklist.md`，逐项判断第一阶段是否真的只需要单服务壳子，还是已经存在必须提前接入的基础设施约束。
3. 阅读 `default-stack.md`，先给出推荐默认方案；如团队尚未确认语言和框架，则落到其中定义的默认后备路线，再判断是否存在必须切换到备选方案的约束。
4. 明确第一阶段目标是最小可运行服务壳子，而不是一次性铺开完整业务逻辑、完整数据模型或微服务拆分。
5. 输出推荐默认方案、备选方案与切换条件，并说明为什么默认采用单服务 + 轻量 REST API 风格。
6. 定义最小服务结构，至少覆盖应用入口、配置加载、基础路由、健康检查、统一错误处理、结构化日志、业务占位和测试目录。
7. 输出首批应创建的目录、模块、文件和初始化步骤，并说明每一项存在的原因。
8. 明确配置、健康检查、错误处理、日志和首批测试的最小边界，避免在 scaffold 阶段把业务细节、鉴权实现或基础设施细节提前固化。
9. 说明与 `backend-development` 的衔接方式，包括哪些占位层后续会被真实业务实现替换。
10. 明确本阶段不应提前做的事，例如微服务拆分、复杂部署编排、完整数据库建模、完整鉴权体系和大规模集成接入。
11. 输出验证方式、未验证项和风险点。

## 输出格式

```text
【当前启动判断】
【推荐默认方案】
【备选方案与取舍】
【首批应创建的目录 / 模块 / 文件】
【初始化步骤】
【最小边界说明】
【与 backend-development 的衔接方式】
【本阶段不应提前做的事】
【下一步建议 Skill】
【验证方式】
【未验证项】
【风险点】
```

## 约束

- 这是服务端新项目骨架层，不替代 `api-design`，也不替代 `backend-development`。
- `scaffold-checklist.md` 和 `default-stack.md` 必须显式读取并作为判断依据使用。
- 第一阶段只创建最小可运行壳子，不默认展开微服务拆分、复杂部署编排或平台化重构。
- 不把完整业务逻辑、完整数据建模、完整鉴权体系或完整基础设施接入一次性铺满。
- 没有明确证据时，不把语言、框架、基础设施或部署偏好写成既定事实。
- 不能声称“已验证”而没有给出验证方式；无法验证的部分必须进入未验证项。

## 不适用场景

- 已有稳定服务端项目，当前只是继续开发或局部改动。
- 当前更需要先做 `api-design`、`technical-design` 或 greenfield 启动分流，而不是直接起服务骨架。
- 第一阶段只是现有服务的小改动，不需要新建项目。
- 需求、接口契约或技术方向仍未确认，无法判断最小服务边界。

## 使用示例

最小调用示例：

```text
请使用 backend-service-scaffold skill，为当前项目创建服务端新项目骨架。

请先阅读：
1. 项目入口文档
2. skills/common/
3. 第一阶段目标
4. skills/backend-service-scaffold/scaffold-checklist.md
5. skills/backend-service-scaffold/default-stack.md

请输出：
1. 推荐默认方案
2. 首批应创建的目录 / 模块 / 文件
3. 初始化步骤
4. 下一步建议 Skill
5. 验证方式
6. 未验证项

要求：
- 只创建最小可运行壳子。
- 不要提前展开完整业务逻辑。
```

更多示例见 `usage-examples.md`。
